# infra.dofe.ai 架构深度审查与优化建议

> 审查日期：2026-06-25
> **实施状态：全部 P0/P1/P2 项已完成** (2026-06-25)
> 详见：[架构实施路线图](./infra-architecture-roadmap-2026-06-25.md)
> 范围：`infra.dofe.ai` monorepo
> 结论：整体架构方向是对的，边界也比旧版清晰，但仍有几处“会放大维护成本”的结构问题，建议按优先级尽快收口。

## 结论先行

当前仓库已经具备较好的基础设施分层：`contracts / utils / common / prisma / redis / rabbitmq / shared-db / clients / shared-services` 的职责是可理解的，`package.json exports` 和 `build-all.sh` 也在把消费者从深层路径里拉回来。  
但从审查结果看，仍有四类问题值得优先处理：

1. 包边界仍存在少量“语义不一致”。
2. 构建/导出策略已经可用，但还不够统一。
3. 兼容别名和历史路径会继续拖慢演进。
4. 文档需要把真实的依赖规则写死，不然会反复回到旧习惯。

---

## P0 需要立刻收口

### 1. 继续清理跨包依赖的”隐性入口” ✅ DONE (Step 1)

现状上，大部分核心包已经声明了依赖，但仓库里仍然能看到通过 `@app/*`、深层 `dist` 路径或历史兼容导入来跨包访问实现细节。  
这类入口短期能跑，长期会让依赖图失真。

建议：

- 继续把跨包引用统一到 `@dofe/infra-*` 的正式导出面。
- 禁止新的 `@app/*` 语义进入 infra 仓库。
- 对必须保留的兼容导入，明确加“过渡期”说明和下线日期。

### 2. `shared-services` 仍然偏”上层聚合包” ✅ DONE (Step 5)

`shared-services` 现在不仅是服务层，还承担了较多组合、适配、桥接职责。  
这本身不算错，但它已经接近“业务基础设施聚合层”，再继续加功能会变成新的大杂烩。

建议：

- 保持 `shared-services` 只负责“依赖业务能力的复合服务”。
- 纯客户端能力继续留在 `clients`。
- 需要复用的业务能力先抽接口，再放实现，避免以后又回到循环依赖。

### 3. `prisma` 的 `db-metrics` 目录层级仍然别扭 ✅ DONE (Step 4)

`db-metrics/src` 这一层嵌套不自然，导出路径也显得不干净。  
这类结构虽然不一定立刻出错，但会持续制造认知噪音。

建议：

- 扁平化 `db-metrics` 目录。
- 同步调整 `exports`，让模块路径和源码目录一致。

---

## P1 重要优化

### 4. `common` 适合再做一次”职责收边” ✅ DONE (Step 5)

`common` 已经很强，但也最容易膨胀。它现在同时承载配置、装饰器、过滤器、守卫、拦截器、ts-rest、通用类型和部分工具。  
这不是马上要拆，但应该开始控制增长。

建议：

- 维持 `common` 为“跨项目通用运行时能力”。
- 非通用、强领域化的东西不要再往里塞。
- 后续新模块优先考虑独立包或至少独立子域目录。

### 5. `utils` 的兼容别名需要明确退场计划 ✅ DONE (Step 1)

仓库里同时保留了 `enviroment.util.ts` 和 `environment.util.ts`，这对兼容有帮助，但也会让新旧引用长期并存。

建议：

- 把新文件名作为唯一推荐入口。
- 旧拼写只保留一段时间的兼容导出。
- 文档里明确“新代码禁止使用旧拼写”。

### 6. 构建脚本已经能用，但规则应写成仓库约束 ✅ DONE (Step 6)

`scripts/build-all.sh` 现在承担了单次编译、分发产物、复制 i18n 资源、生成 exports 等职责。这个方案可行，但已经不是“简单脚本”了。  
它实际上是仓库的构建契约。

建议：

- 在 README 里补上“为什么必须走根构建”的说明。
- 把构建步骤拆成可读的小步骤，减少后续维护时误改。
- 为关键步骤加最小化校验，比如缺少 `dist` 时直接失败而不是静默跳过。

### 7. 依赖声明要继续向”真实依赖”收敛 ✅ DONE (Step 2)

现在各包 `dependencies` / `peerDependencies` 的方向基本合理，但仍要避免靠 hoist、workspace 假设或历史路径吃饭。

建议：

- 只要源码 import 了别的包，就在包清单里显式声明。
- 能作为 peer 的依赖就不要偷偷落到实现里。
- 对运行时必须存在的依赖，优先声明为 `dependencies`。

## P2 建议完善

### 8. 根 README 还可以更像”架构合同” ✅ DONE (Step 7)

现在 README 已经有目录和使用示例，但对“怎么依赖、怎么构建、怎么导入、禁止什么”还不够硬。

建议补三块内容：

- 包分层图。
- 正式导入路径示例。
- 兼容/废弃路径清单。

### 9. 建议补一份”仓库级 ADR / 架构说明” ✅ DONE (Step 7)

这类 infra 仓库最怕反复重做边界。  
建议把以下规则固定下来：

- 哪些包允许依赖哪些包。
- 哪些能力必须走子路径导出。
- 哪些目录是稳定 API，哪些只是内部实现。

### 10. `exports` 很强，但要控制增长 ✅ DONE (Step 3)

当前大量包已经走了显式 exports，这很好。  
下一步不建议继续无节制开放更深层模块，除非它真的是稳定 API。

建议：

- 根导出负责常用入口。
- 子路径导出只开放稳定、可长期承诺的模块。
- 临时模块和内部工具不要公开出去。

---

## 推荐落地顺序

1. 先收口跨包依赖和历史兼容入口。
2. 再扁平化 `db-metrics` 这类明显不顺的目录。
3. 然后补齐 README 和架构说明。
4. 最后再考虑把 `common`、`shared-services` 的边界继续细化。

---

## 验证标准

建议用下面几个标准判断优化是否真正完成：

- 新包引用不再依赖 `@app/*` 语义。
- 任意单包都能依靠自身 `package.json` 解释运行时依赖。
- 构建产物路径与源码目录一致。
- README 能让新接手的人在 10 分钟内知道该怎么用、怎么避坑。

---

## 一句话建议

这套 infra 的问题已经不是“能不能用”，而是“边界能不能继续保持干净”。下一阶段最值得做的，不是再堆能力，而是把已有能力的导入面、依赖面和兼容面收紧。
